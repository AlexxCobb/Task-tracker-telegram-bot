# 📋 Task Tracker Telegram Bot

Production-ready Telegram-бот для личного управления задачами, написанный на Java 21 и Spring Boot 3.5. 

Поддерживает простые задачи, иерархические задачи с подзадачами и списки покупок — всё управляется через удобную inline-клавиатуру. Состояние диалога хранится в PostgreSQL, что обеспечивает многошаговые сценарии между сессиями.

---

## ✨ Возможности

- **Умное приветствие** — персонализированный ответ на `/start` для новых и вернувшихся пользователей
- **Три типа задач** — простые задачи, задачи с подзадачами и списки покупок
- **Гибкая фильтрация** — просмотр задач: Все / Активные / Завершённые
- **Полный жизненный цикл** — создание, переименование, отметка выполненных, удаление
- **Массовая очистка** — удаление всех завершённых задач одним действием
- **Контекстная навигация** — кнопка «Назад» в inline-клавиатуре помнит, откуда пришёл пользователь
- **Персистентное состояние диалога** — разговоры переживают перезапуск бота
- **Понятные сообщения об ошибках** — никаких стектрейсов, только человекочитаемые тексты

## 🏗 Архитектура

Проект построен по **слоистой архитектуре** (`bot → service → dao`) с двумя ключевыми паттернами:

**Цепочка обязанностей — Update Dispatcher**

Входящие Telegram-обновления попадают в `UpdateDispatcher`, который перебирает список бинов `UpdateHandler`. Каждый обработчик объявляет метод `canHandle(update)` и обрабатывает обновление только в случае соответствия. Это отделяет логику маршрутизации от бизнес-логики и упрощает добавление новых команд.

```
Telegram Update
      │
      ▼
UpdateDispatcher
      │
      ├─► CommandHandler     (canHandle → true)  ──► handle()
      ├─► CallbackHandler    (canHandle → false)
      └─► MessageHandler     (canHandle → false)
```

**Машина состояний диалога**

Контекст разговора каждого пользователя представлен значением enum `DialogState`, хранящимся в PostgreSQL. Обработчики читают и переключают состояние, обеспечивая многошаговые сценарии (например, «введи название → подтверди → сохрани») без привязки к сессии в памяти.

**Обработка ошибок**

Централизованный `ExceptionHandler` перехватывает все исключения на уровне бота и отправляет пользователю понятные сообщения вместо технических деталей.

## ⚙️ Как это работает

```
Пользователь отправляет сообщение или нажимает кнопку
          │
          ▼
    Серверы Telegram
          │  (long polling)
          ▼
   UpdateDispatcher
          │
          │  перебирает List<UpdateHandler>
          │  вызывает canHandle(update) у каждого
          │
          ▼
   Подходящий UpdateHandler
          │
          ├── читает DialogState из БД
          ├── вызывает TaskService / UserService
          ├── при необходимости переключает DialogState
          └── отправляет ответ через TelegramBots API
```

Все переходы состояний сохраняются синхронно до отправки ответа, что гарантирует консистентность даже при перезапуске процесса в середине диалога.
